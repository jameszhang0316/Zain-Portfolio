## Executive summary

This project is a **Create React App (CRA)** portfolio template (“masterPortfolio”) built with **React 16**. Content is primarily configured via `src/portfolio.js` and styling via `styled-components` themes in `src/theme.js`. An optional **GitHub GraphQL fetcher** (`git_data_fetcher.mjs`) pulls pull requests, issues, contributed orgs, and pinned repos using a token in `.env`, then writes JSON into `src/shared/opensource/` which the UI renders under the “Open Source” / “Projects” sections.

## Tech stack

- **Frontend**: React 16 (`react`, `react-dom`) + `react-router-dom` for routing
- **Build tooling**: CRA via `react-scripts@3.2.0`
- **Styling**:
  - `styled-components` (`ThemeProvider`, global styles in `src/global.js`)
  - `baseui` + `styletron` provider setup in `src/index.js` (note: most components appear to rely on CSS + styled-components theme variables rather than Base Web primitives)
- **Animations / UI**: `react-reveal`, Font Awesome, Iconify script
- **Charts**: `chart.js@2.x`, `react-chartjs-2@2.x`
- **Resume viewer**: `react-pdf`, `pdfjs-dist`
- **GitHub data**: `node-fetch@2.x`, GitHub GraphQL API via `git_data_fetcher.mjs`

## Key scripts and workflows

From `package.json`:

- **`npm start`**: `react-scripts --openssl-legacy-provider start`
- **`npm run build`**: `react-scripts --openssl-legacy-provider build`
- **Deploy**: `npm run deploy` publishes `build/` to `gh-pages` branch using `gh-pages`

Notes:

- The `--openssl-legacy-provider` flag is present in both build and start. This is typically used to work around older webpack/crypto issues on some Node versions; with Node 20 it may or may not still be required depending on the dependency tree.

GitHub Actions:

- **`.github/workflows/nodejs.yml`**: runs `npm ci`, `npm run build --if-present`, and `npm test` on pushes/PRs to `master` and `releases/*`, using Node 20.x.
- **`.github/workflows/pullRequestsDataFetcher.yml`**: runs a GraphQL query action, but appears misconfigured (details below).

## App runtime flow (what happens on load)

1. `src/index.js` renders `App` into `#root`.
2. `src/App.js` selects `chosenTheme` from `src/theme.js` and passes it down:
   - `ThemeProvider` for styled-components
   - `Main` receives `theme={chosenTheme}` prop
3. `src/containers/Main.js` sets a CSS variable `--scrollbar-color` from `theme.imageHighlight`.
4. Routing:
   - `/` renders `Splash` if `settings.isSplash === true`, else `Home`
   - Other pages render as expected (Open Source page composes orgs + charts + PRs + issues)

## Open Source / GitHub data pipeline

### Configuration

- **`env.example`** expects:
  - `GITHUB_TOKEN=...`
  - `GITHUB_USERNAME=...`
- Local usage described in `README.md`: create `.env` and run:
  - `node git_data_fetcher.mjs`

### What `git_data_fetcher.mjs` does

It makes GitHub GraphQL API calls (POST `https://api.github.com/graphql`) and writes results into:

- `src/shared/opensource/pull_requests.json` (also computes open/closed/merged counts)
- `src/shared/opensource/issues.json` (open/closed counts)
- `src/shared/opensource/organizations.json` (unique org owners from contributed repos)
- `src/shared/opensource/projects.json` (pinned repos with a curated language→icon mapping)

### How the UI consumes it

Examples:

- `src/containers/pullRequests/PullRequests.js` imports `../../shared/opensource/pull_requests.json`
- `src/containers/issues/Issues.js` imports `../../shared/opensource/issues.json`
- `src/containers/organizations/Organizations.js` imports `../../shared/opensource/organizations.json`
- `src/pages/projects/Projects.js` imports `../../shared/opensource/projects.json`

Operationally: the portfolio **does not fetch GitHub data client-side** at runtime; it **renders static JSON bundled at build time**. To refresh it, re-run the fetcher and rebuild.

## Docker setup

- `Dockerfile`:
  - Uses `node:20.11-alpine`
  - Copies `package.json` + `package-lock.json`, runs `npm install`
  - Copies rest of repo and runs `npm start`
  - `EXPOSE 3001`
- `docker-compose.yaml`:
  - Maps `3001:3000` and runs the app with bind mounts (`.:/app`)

Notes / potential issues:

- CRA’s default dev server listens on **3000**; the compose mapping (`3001:3000`) makes sense for host port 3001.
- `Dockerfile`’s `EXPOSE 3001` is likely a mismatch (should typically be `EXPOSE 3000` for CRA dev server), though `EXPOSE` is only informational.

## CI/CD observations

### Node.js CI workflow

The `nodejs.yml` workflow runs tests (`npm test`) with `CI=true`, which in CRA causes tests to fail if there are failing tests or if the test runner expects watch mode behavior differences.

### Pull request data fetcher workflow (likely broken)

`.github/workflows/pullRequestsDataFetcher.yml` issues:

- Trigger is `push` to branch `all` (a branch named `all` must exist; otherwise it won’t run)
- The final echo references `steps.get_latest_release.outputs.data`, but the step id is `get_pull_requests`, and the action output variable in that echo does not match any prior step. This makes the logging step incorrect as written.

If this workflow is intended to generate/update `src/shared/opensource/*.json` automatically, it currently does not implement writing artifacts or committing changes.

## Code quality / maintainability notes

- **Pinned upstream identity**: Many strings are still “Ashutosh” throughout `src/portfolio.js` and `public/index.html`. If this is meant to be “Zain”’s portfolio, these need updating.
- **Third-party tracking/ads**: `public/index.html` includes Adsense and Analytics IDs. This is a privacy and compliance consideration; remove if not explicitly required.
- **Dependency age**: React 16 + CRA 3.2 are quite old by 2026 standards. This affects:
  - Security patch surface
  - Compatibility with modern tooling
  - Build warnings and Node/OpenSSL workarounds
- **Data refresh workflow**: GitHub data is static JSON. If you need it to stay current without manual steps, consider automating fetch+commit on a schedule or switching to runtime API calls (with care for token secrecy).
